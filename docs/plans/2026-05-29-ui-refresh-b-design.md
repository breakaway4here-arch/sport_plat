# sport_plat UI Refresh Design

## Objective

把 `sport_plat` 的手机端界面重构成“方案 B | 数据习惯型”的风格：清爽、稳定、偏数据工具，不做营销页，不做重插画，不牺牲现有训练与打卡逻辑。

这个改造的核心不是换皮，而是把当前产品统一成一套可长期使用的移动端健康工具界面。用户每天打开时，应该一眼看到今天该做什么、完成了多少、还有什么没做。

## Design Direction

- 参考图只作为气质来源，不照抄版式
- 主风格：白底、蓝色主导、少量绿色完成态、轻阴影、浅边框
- 信息气质：理性、轻量、清晰、偏工具
- 内容优先级：计划 > 今日训练 > 打卡记录 > 数据统计
- 移动端优先：主要适配 320px 到 430px 宽度

## Visual System

### Color Tokens

- `--primary`: `#2563EB`
- `--primary-dark`: `#1D4ED8`
- `--primary-soft`: `#EAF2FF`
- `--success`: `#22C55E`
- `--success-soft`: `#EAF8EF`
- `--bg`: `#F6F8FC`
- `--card`: `#FFFFFF`
- `--border`: `#D8E0EE`
- `--text`: `#111827`
- `--text-secondary`: `#6B7280`
- `--text-tertiary`: `#9CA3AF`

Rules:

- 蓝色是主色，只用于主按钮、选中态、进度、当前 tab、关键强调
- 绿色只用于完成、打卡成功、正反馈，不作为主导航色
- 不使用大面积紫色、橙棕色、深黑底、暖纸色
- 不使用强渐变背景，不使用明显彩色装饰 orb

### Typography

- 保持系统无衬线中文字体
- 标题更短更重，按钮文字更直接
- 数字信息优先，说明文字次之，辅助信息最弱
- 不引入花哨 display font

### Shape and Depth

- 卡片圆角：8px 到 12px
- 按钮圆角：8px 左右
- 阴影：极轻，优先细边框感
- 列表分隔：浅边框，不用厚重阴影做分层

### Layout Rhythm

- 页面边距：12px 到 16px
- 卡片间距：8px 到 12px
- 模块内部间距：6px 到 10px
- 所有关键按钮保持至少 44px 触控高度

## Component Rules

### Tabs

- 底部固定 tab
- 选中态用蓝色
- 文案只保留 `计划 / 训练 / 打卡 / 统计`
- tab 尺寸必须适配 320px 屏幕，不挤压成不可点区域

### Cards

- 统一使用白底卡片
- 标题在左，辅助动作在右
- 卡片内信息按“标题 - 核心信息 - 辅助信息”排序
- 避免一张卡里再套一张卡

### Chips

- 默认白底浅边框
- 选中态为蓝底白字
- 单选 chips 和多选 chips 使用同一套视觉语言
- chip 必须保证可点击面积，不做小字贴边

### Buttons

- 主按钮：蓝底白字
- 次按钮：白底蓝边
- 轻量按钮：只在编辑、替换、删除等次级操作使用
- 删除类按钮使用文本或轻量图标，不做夸张强调

### Modals

- 底部上滑弹窗
- 宽度适配手机屏
- 内容高于 70vh 时允许内部滚动
- 关闭按钮要明显，触控区足够大

### Progress / Data

- 进度条细而清晰
- 统计数字块优先显示“数值 + 单位”
- 图表只保留必要骨架，不做复杂装饰

## Page Requirements

### 1. 计划页

目标：让用户快速生成或修改训练计划。

要求：

- 训练目标、训练天数、时长、器械分组清楚
- 生成按钮始终可恢复，不能因为切 tab 丢失
- 已有计划时，回到计划页必须能继续修改
- “自定义动作”作为次级入口存在，但不能削弱主流程
- 计划结果区域需要清晰展示当前计划，同时支持重新生成、替换动作

验收：

- 生成后切换 tab 再回来，按钮仍可见
- 点击“修改条件”能恢复编辑状态
- 点击“重新生成”只替换计划内容，不丢约束

### 2. 今日训练页

目标：今天要做什么、做了多少、能不能打卡，一眼看清。

要求：

- 顶部有今日目标摘要
- 中间为动作列表，动作项要紧凑但不拥挤
- 底部必须有明确打卡按钮
- 额外项目可以在今天添加，但不能污染计划完成进度
- 如果当天没有计划，显示休息或空状态

验收：

- 额外项目在未打卡时只作为临时项存在
- 打卡后，额外项目和计划动作都能正确保存到今日记录
- 已打卡后再添加额外项目，时长会追加到已存在记录

### 3. 打卡记录页

目标：让用户快速查看最近的完成情况。

要求：

- 月历和列表保留
- 重点展示日期、完成动作和总时长
- 页面视觉要比当前更轻，不要压迫
- 记录中若存在未知动作 id，也不应报错

验收：

- 月历日期可点击
- 记录列表展示正常
- 已删除自定义动作的历史记录仍能显示

### 4. 统计页

目标：看见趋势，而不是堆数值。

要求：

- 仅保留有用的统计卡片
- 数据块以蓝色为主，完成态或正向指标可用绿色
- 图表简洁，减少无意义装饰
- 数字、单位、说明层级分明

验收：

- 本周 / 本月训练天数和时长清晰
- 连续训练天数清晰
- 部位分布和总览信息可读

## Behavior Rules

- 不改变现有 localStorage 数据结构，除非为了兼容性必须增加字段
- 不删除用户已有计划、打卡、自定义动作
- 不让 service worker 缓存覆盖新资源版本
- 任何 UI 改造都不能引入破坏现有生成、替换、打卡、统计逻辑的行为回归

## Testing Requirements

必须保持并扩展现有浏览器流程测试，至少覆盖：

- 生成计划
- 计划页切 tab 后恢复编辑状态
- 修改条件后可重新生成
- 替换单个动作
- 自定义动作可进入目标/器械筛选
- 自定义输入 HTML 转义和默认值兜底
- 今日未打卡时添加额外项目不应提前生成 check-in
- 仅额外项目也可以打卡
- 已打卡后继续添加额外项目会追加到已有记录
- 休息日显示空状态
- 历史页兼容未知动作 id

## Implementation Steps

### Phase 1. Design Tokens

1. 统一 CSS 变量：主色、背景、边框、文本、完成态
2. 收敛圆角、阴影、间距和按钮高度
3. 保证 320px 下布局不挤压

### Phase 2. Shared Components

1. 重做 tab bar 样式
2. 统一 card、chip、button、modal、input
3. 统一动作卡片和统计卡片的层级

### Phase 3. Plan Page

1. 调整计划生成器布局
2. 保证生成后仍能回到编辑态
3. 让计划结果卡更像“当前计划摘要”，而不是纯列表

### Phase 4. Today Page

1. 让今日目标、动作列表、打卡按钮成为首屏中心
2. 额外项目改成附加项，不污染计划进度
3. 保留完整的打卡逻辑和追加逻辑

### Phase 5. History and Stats

1. 历史页轻量化
2. 统计页保留核心数据卡和图表
3. 修整空状态与异常状态

### Phase 6. Verification

1. 跑浏览器流程测试
2. 手机视口检查 390px 和 320px
3. 确认切 tab、修改计划、打卡、统计都无回归

## Acceptance Criteria

- 界面整体明显靠近方案 B 的清爽数据风格
- 手机端 320px 到 430px 可用
- 计划页切换 tab 后不丢编辑入口
- 训练页额外项目逻辑不再污染计划完成度
- 现有测试与新增测试全部通过
- 不新增会破坏用户数据的行为

## Notes

- 这份设计文档只描述 UI 和交互落地方向，不改业务语义
- 如果后续要进一步收口，可以再拆成一个更细的实施计划
